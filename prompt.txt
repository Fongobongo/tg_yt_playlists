1. **Tech Stack**:

2. **Bot Functionality**:
   - Each Telegram chat (group or private) is a separate "session". Users in different chats are isolated.
   - Command `/start`: initializes the session for the chat and sends a welcome message with instructions.
   - When a user sends a YouTube playlist URL (e.g., `https://www.youtube.com/playlist?list=...`), the bot must:
     * Validate the URL and extract the playlist ID.
     * Use yt-dlp to fetch the playlist metadata (list of videos). Run yt-dlp in a thread executor to avoid blocking the event loop.
     * For each video, obtain the YouTube video ID, title, and URL (https://www.youtube.com/watch?v=<id>).
     * Store the playlist and its videos in the database, linked to the current session and the user (by Telegram ID and username).
     * After storing, compute the intersection of video IDs across *all* playlists that have been added in this session (including the new one). Videos that appear in every playlist are considered common.
     * Reply to the user with a formatted list of common videos: each line should include the video title and a clickable link. If no common videos exist, reply with a friendly message saying so.
   - Optional: a command `/playlists` to list all playlists in the session, and `/clear` to remove all data for the session (for admins only). Not required but nice to have.

3. **Database Schema** (PostgreSQL):
   - `sessions`:
     * `id` UUID primary key
     * `chat_id` BIGINT unique (the Telegram chat identifier)
     * `created_at` TIMESTAMPTZ default now()
   - `users`:
     * `id` UUID primary key
     * `session_id` FK references sessions(id) (ON DELETE CASCADE)
     * `telegram_id` BIGINT (the user's Telegram ID within the session; combined with session_id should be effectively unique)
     * `username` TEXT (Telegram username, optional)
     * `created_at` TIMESTAMPTZ default now()
   - `playlists`:
     * `id` UUID primary key
     * `session_id` FK references sessions(id)
     * `user_id` FK references users(id) (ON DELETE CASCADE)
     * `youtube_playlist_id` TEXT (the YouTube playlist ID, e.g., PL...)
     * `title` TEXT (playlist title)
     * `url` TEXT (full playlist URL)
     * `created_at` TIMESTAMPTZ default now()
   - `videos`:
     * `id` UUID primary key
     * `playlist_id` FK references playlists(id) (ON DELETE CASCADE)
     * `youtube_video_id` TEXT (YouTube video ID)
     * `title` TEXT (video title)
     * `url` TEXT (video URL)
     * `position` INT (order in playlist, optional but yt-dlp provides)
     * `created_at` TIMESTAMPTZ default now()
   - Indexes: on `users(session_id, telegram_id)`, on `playlists(session_id)`, on `videos(playlist_id)`, and maybe on `videos(youtube_video_id)` for faster intersection queries.

   We'll use raw asyncpg for simplicity; no ORM. We'll write SQL queries directly.
